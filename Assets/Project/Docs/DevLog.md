## 2026-06-02

## 오늘 목표
- Unity 6 엔진의 기본 인터페이스와 핵심 기능들을 빠르게 복습한다.
- 포트폴리오 개발에 들어가기 전 Unity 작업 환경과 기본 용어를 다시 정리한다.

### 오늘 한 일
- <<Do it! 유니티 6로 게임 제작 & 출시하기>>를 2-2까지 읽으며 Unity 기본 구조를 복습했다.
- Unity Hub, 프로젝트 실행, Scene View, Game View, Hierarchy, Inspector, Project 창의 역할을 확인했다.
- 2D GameObject와 3D GameObject의 기본 차이를 간단히 살펴보았다.
- GameObject와 Component의 관계를 다시 정리했다.
- URP와 HDRP의 차이를 간단하게 정리했다.
- Camera 설정이 Scene 화면과 Game 화면에 어떤 영향을 주는지 확인했다.
- 2D 프로젝트에서 사용할 수 있는 Light 2D 설정을 간단히 확인했다.
- 오늘은 기능 구현보다는 Unity 엔진에 다시 익숙해지는데 집중했다.

### 배운 내용
- Unity의 GameObject는 씬에 배치되는 기본 단위이며, 실제 기능은 Componet를 추가해 구성한다.
- Inspector는 선택한 오브젝트의 Transform, Sprite Renderer, Camera, Light 같은 컴포넌트 값을 확인하고 수정하는 핵심 창이다.
- Hierachy는 현재 Scene에 존재하는 오브젝트 구조를 확인하는 곳이고, Project창은 Assets 폴더 안의 파일과 리소스를 관리하는 곳이다.
- 2D GameObject는 Sprite 기반의 오브젝트를 다루는데 적합하고, 3D GameObject는 Mesh와 3D 공간을 기반으로 구성된다.
- Camera 설정은 실제 플레이어가 보게 되는 Game View에 직접적인 영향을 주기 때문에, 2D 탑다운 액션 로그라이트에서도 초반부터 기준을 잘 잡아야 한다.
- Light 2D는 2D 환경에서도 분위기와 가독성을 조절할 수 있는 기능이며, 추후 스테이지 연출이나 피격 효과 등에 응용할 수 있다.

### 포트폴리오에 적용할 부분
- Player, Enemy, Camera, UI를 각각 GameObject 단위로 구성하고, 기능은 Component와 Script로 분리하는 방향으로 설계한다.
- 앞으로 PlayerMovement, Health, EnemyAI, CameraFollow 같은 스크립트를 각각의 책임에 맞게 나누어 작성한다.
- 프로젝트 폴더 구조를 미리 정리해 Scripts, Prefabs, Scenes, Sprites, UI 리소스를 구분해서 관리한다.
- Camera는 플레이어 추적 기능을 구현할 때 중요한 기준 오브젝트가 되므로, 다음 작업에서 Camera 설정과 추적 방식을 함께 확인한다.

### 해결한 방법

- 구현은 무리하게 진행하지 않고, 다음 작업에서 플레이어 오브젝트 생성과 이동 구현으로 이어가기로 했다.

### 오늘 목표

### 오늘 한 일
- URP 2D 프로젝트를 생성하고 기본 게임 화면을 설정했다.
- Game View에서 해상도를 설정해 실제 게임 화면 기준을 확인했다.
- 외부 리소스를 가져와 이미지에 맞게 Pixel Per Unit 값을 조정했다.
- Tile Palette를 이용해 직접 타일을 배치하며 간단한 맵을 만들어보았다.
- Tilemap에 충돌 처리를 적용하는 방법을 학습했다.
- TextMeshPro를 활용해 현재 스테이지를 표시하는 텍스트 UI를 생성했다.
### 배운 내용
- Game View의 해상도 설정은 실제 플레이 화면 기준을 확인하는 데 중요하다.
- Pixel Per Unit 값은 이미지 리소스가 Unity 월드에서 어느 정도 크기로 보일지 결정하는 요소이므로, 리소스의 픽셀 크기에 맞게 조정해야 한다.
- Tile Palette를 사용하면 2D 맵을 타일 단위로 빠르게 배치할 수 있다.
- Tilemap에 Collider를 적용하면 플레이어가 벽이나 장애물 타일을 통과하지 못하도록 충돌 영역을 만들 수 있다.
- Composite Collider 2D를 함께 사용하면 연결된 타일들의 충돌 영역을 하나의 연속된 충돌 구조로 합성할 수 있다.
- 이를 통해 타일별로 나뉜 충돌 경계를 줄이고, 충돌 처리를 더 효율적으로 관리할 수 있다.
- TextMeshPro는 일반 텍스트보다 가독성과 표현력이 좋아 UI 텍스트를 구성할 때 유용하다.
### 포트폴리오에 적용할 부분
- 포트폴리오 프로젝트의 스테이지는 Tilemap 기반으로 구성할 수 있을 것 같다.
- 초반에는 직접 Tile Palette로 테스트 맵을 만들고, 이후에는 절차적 맵 생성 방식을 적용할 수 있는지 조사해볼 예정이다.
- 길찾기나 절차적 생성 알고리즘을 참고하여, 에디터에서 타일을 하나하나 찍지 않아도 일정한 규칙에 따라 맵을 생성하는 구조를 만들어보고 싶다.
- 예를 들어 방, 길, 벽 타일을 코드로 배치하고 Tilemap에 자동으로 그려주는 방식으로 확장할 수 있을 것 같다.
- 스테이지 번호나 현재 웨이브 정보는 TMP를 활용해 UI에 표시할 수 있을 것 같다.
### 막힌 부분
- 오늘은 컨디션 저하로 인해 플레이어 이동 구현까지는 진행하지 못했다.
- Tilemap, 리소스 설정, 충돌 처리, TMP UI 등 새로 확인할 내용이 많아 구현 범위를 줄였다.
### 해결한 방법
- 오늘은 무리해서 플레이어 이동까지 진행하지 않고, 2D 맵 구성과 UI 표시 흐름을 학습하는 데 집중했다.
- 대신 포트폴리오에 적용할 수 있는 아이디어를 정리해두어 이후 작업으로 이어갈 수 있도록 했다.
### 내일 할 일

## 2026-06-05

### 오늘 목표
- Enemy 오브젝트 생성 방법 학습
- Tilemap 범위 기반 Enemy 스폰 위치 설정 학습
- Player와 Enemy의 공통 스탯 구조 학습
- Player와 Enemy를 따라다니는 체력바 UI 구현 방식 학습
### 오늘 한 일
- Enemy 오브젝트를 생성하고 스폰 위치를 설정하는 방법을 학습했다.
- Tilemap의 Bounds 크기를 기준으로 맵 내부 좌표를 탐색하고, 그중 스폰 가능한 위치만 선별하여 리스트에 저장했다.
- 저장된 스폰 가능 위치 리스트에서 Random.Range를 사용해 임의의 인덱스를 선택하고, 해당 위치에 Enemy가 생성되도록 구성했다.
- Player와 Enemy가 공통으로 사용할 기본 스탯을 EntityState 구조체로 정의했다.
- 공통 스탯과 기본 동작을 관리하기 위해 EntityBase 클래스를 만들고, Player와 Enemy가 이를 상속받는 구조를 학습했다.
- UI Image를 활용해 Player와 Enemy를 따라다니는 체력바 UI를 만드는 방법을 학습했다.
### 배운 내용
- Tilemap의 Bounds 정보를 활용하면 맵 범위 안에서 오브젝트를 생성할 수 있는 후보 위치를 계산할 수 있다.
- 스폰 가능한 위치를 미리 리스트로 저장해두면, Enemy 생성 시 매번 위치를 새로 계산하지 않고 랜덤 인덱스만 선택해서 사용할 수 있다.
- Player와 Enemy처럼 공통 스탯이 필요한 객체는 별도의 구조체와 베이스 클래스를 통해 중복 코드를 줄일 수 있다.
- EntityBase 같은 부모 클래스를 두면 체력, 이동속도, 공격력 같은 공통 데이터를 한 곳에서 관리할 수 있다.
- 월드 공간에 존재하는 캐릭터 위에 UI 체력바를 표시하려면, 대상의 위치를 기준으로 UI 위치를 갱신해야 한다.
### 막힌 부분
- Tilemap의 Bounds를 기준으로 스폰 가능한 위치를 선별하는 흐름이 처음에는 복잡하게 느껴졌다.
- Player와 Enemy의 기본 스탯을 각각 따로 만들지 않고 공통 구조로 묶는 방식이 아직 완전히 익숙하지 않았다.
- 체력바 UI가 실제 캐릭터를 따라다니도록 연결하는 좌표 처리 방식이 헷갈릴 수 있다고 느꼈다.
### 해결한 방법
- Tilemap 범위 안의 좌표를 순회하면서 스폰 가능한 위치만 리스트에 저장하는 방식으로 흐름을 나누어 이해했다.
- EntityState 구조체에는 기본 스탯 데이터를 두고, EntityBase 클래스에는 Player와 Enemy가 공통으로 사용할 기능을 두는 방식으로 역할을 구분했다.
- 체력바 UI는 대상 오브젝트의 위치를 기준으로 계속 갱신되도록 만들어, 캐릭터를 따라다니는 방식으로 이해했다.
### 내일 할 일
- Enemy가 Player를 대상으로 기본 공격을 수행하는 흐름 학습
- Player의 공격 대상 탐색 및 목표 설정 방식 학습
- 공통 스탯 관리를 위한 스탯 클래스 구조 학습
- 기본 공격 스킬을 생성하고 캐릭터의 공격 로직과 연결하는 방법 학습

### 오늘 한 일
- 적이 일정한 공격 쿨타임 간격으로 총알을 발사해 플레이어를 공격하도록 구현했다.
- 총알의 충돌 판정은 OnTriggerEnter2D를 사용해 처리했다.
- 충돌한 대상의 태그가 Player인지 확인한 뒤, TryGetComponent로 EntityBase 컴포넌트를 가지고 있는지 검사했다.
- 조건을 만족하는 경우 TakeDamage를 호출해 플레이어에게 데미지가 적용되도록 만들었다.
### 배운 내용
- OnTriggerEnter2D를 활용하면 총알처럼 충돌 시 특정 동작을 실행해야 하는 오브젝트의 판정을 처리할 수 있다는 것을 학습했다.
- 단순히 태그만 검사하는 것이 아니라, TryGetComponent를 함께 사용하면 대상이 실제로 데미지를 받을 수 있는 구조인지 한 번 더 확인할 수 있다는 점을 배웠다.
- enum으로 정의한 타입을 .ToString()으로 문자열화하여 코루틴 제어에 활용하는 방식이 인상 깊었다.
- 적 공격, 총알 충돌, 데미지 적용 흐름을 분리하면 이후 플레이어 공격이나 스킬 시스템으로 확장하기 쉬운 구조가 된다는 점을 확인했다.
### 막힌 부분

- 스킬 시스템의 전체 동작 흐름을 코드 기준으로 파악했다.
    - SkillTemplate은 스킬 데이터를 저장하는 역할을 한다는 점을 정리했다.
    - buffStatList를 통해 스킬이 영향을 줄 스탯만 선택적으로 관리할 수 있다는 점을 확인했다.
    - SkillBuff.OnLevelUp()에서 스킬 데이터에 저장된 증가량을 owner의 실제 Stat의 BonusValue에 적용하는 흐름을 확인했다.
- SkillSystem, SkillGad, ProjectileGad의 주요 코드 흐름을 살펴봤다.
    - SkillSystem의 skills는 습득 가능하거나 강화 가능한 전체 스킬의 런타임 객체를 관리하는 Dictionary로 이해했다.
    - StartCoroutine(nameof(SpawnProjectile))에서 nameof가 메서드 이름을 문자열로 변환하는 역할임을 확인했다.
    - LayerMask.NameToLayer("Enemy")와 비트 시프트를 통해 Enemy 레이어만 검사하는 LayerMask를 만드는 방식을 정리했다.
### 배운 내용
- Stat 클래스는 능력치를 단순한 숫자로 관리하는 것이 아니라, 기본값과 보정값을 분리하여 최종값을 계산하기 위한 구조다.
- Value는 DefaultValue와 BonusValue가 반영된 최종값이므로, 버프나 성장으로 변할 수 있는 능력치의 기준값으로 사용된다.
- CurrentHP와 HP는 모두 체력과 관련되어 있지만 성격이 다르다.
    - CurrentHP는 현재 남아있는 체력 상태값이다.
    - HP는 현재 체력이 넘지 말아야 할 최대 기준값이다.
- SkillTemplate은 스킬의 고정 데이터를 저장하고, SkillBuff는 해당 데이터를 바탕으로 owner의 실제 스탯에 보정값을 적용한다.
- 스킬 시스템은 캐릭터의 값을 직접 무작정 바꾸는 것이 아니라, Stat 구조를 통해 능력치에 영향을 주는 방식으로 동작한다.
- LayerMask는 레이어 번호 자체가 아니라, 특정 레이어를 검사하기 위한 비트마스크 값으로 사용된다.
### 막힌 부분
- Stat 구조에서 DefaultValue와 Value가 각각 어떤 상황에서 사용되는지 헷갈렸다.
- SkillBuff.OnLevelUp()에서 buffStatList의 Stat 값이 실제로 어느 대상의 BonusValue에 적용되는지 헷갈렸다.
- SkillSystem의 skills가 습득한 스킬 목록인지, 전체 스킬 목록인지 구분이 필요했다.
- nameof와 LayerMask 비트 시프트처럼 C# 문법과 Unity API가 함께 사용된 부분의 의미를 확인할 필요가 있었다.
### 해결한 방법
- Stat의 역할을 먼저 큰 틀에서 다시 정리한 뒤, DefaultValue, BonusValue, Value의 관계를 순서대로 복습했다.
- SkillTemplate, SkillBuff, SkillSystem의 역할을 분리해서 정리했다.
    - SkillTemplate은 스킬 데이터 원본이다.
    - SkillBuff는 스킬 데이터에 저장된 값을 owner의 Stat.BonusValue에 적용한다.
    - SkillSystem은 스킬 런타임 객체와 레벨 상태를 관리한다.
- nameof는 문자열 기반 코루틴 호출에서 메서드 이름을 안전하게 가져오기 위한 방식으로 이해했다.
- LayerMask는 특정 레이어 번호를 비트마스크로 변환하여 Physics2D 검색 필터로 사용하는 구조로 정리했다.
